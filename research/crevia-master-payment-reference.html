On Bcon: No active developer documentation, API reference, or community activity found as of March 2026. Building a payments integration on an unmaintained or defunct service is a critical dependency risk — if the service goes offline, your users cannot pay. Do not use Bcon as a production dependency. NexaPay occupies the same "no-KYC, non-custodial, email-only signup" niche and has verifiable current uptime, an active API, and Base USDC support.

On Helio: Helio was acquired by MoonPay. Its embeddable checkout widget runs on Solana, not EVM chains. For a Base/USDC stack, Helio's React SDK will not work without significant modification. MoonPay Commerce (the rebrand) supports Base via API, but it is a heavier integration than a drop-in widget. This is a v2 consideration, not v1.

III

The "Metadata Trick" — What Actually Works

"Embed a unique ID in the transaction's data field." — This works for raw ETH sends. It does not work for USDC. — The one thing to get right before writing any payment code

The pro-tip as described applies only to raw ETH transfers. When you send ETH directly, the transaction has a data field that can hold arbitrary bytes — you can encode a user ID there. Your backend then reads tx.input to extract it.

USDC is an ERC-20 token. A USDC transfer is a function call: transfer(address to, uint256 amount). The ABI is fixed — there is no extra slot for arbitrary data. If you call writeContract with the standard erc20Abi, you cannot append a user ID to it. Attempting to add extra bytes will cause the call to revert.

Instead, use one of the three correct approaches below.

Approach How it works Complexity For Crevia v1?
Wallet address correlation Create a server-side intent that stores { userId, planId, walletAddress } before the payment. When USDC arrives FROM that wallet, look it up by sender address. No on-chain metadata needed at all. Low — already in the plan ✓ Recommended. Already implemented in /api/payment/intent.
Unique deposit address per user Derive a unique sub-address for each user (HD wallet path). Any USDC sent to that address = identified as that user's payment. Re-aggregate funds periodically. Medium — HD wallet library Overkill for v1. Consider for high volume.
Wrapper smart contract Deploy a thin contract with a pay(bytes32 intentId) function. Emits a custom event with your metadata. Calls USDC.transferFrom internally. High — contract + audit V3+ feature. Not needed for SaaS subscriptions.
Raw ETH with encoded data Send a tiny amount of ETH (e.g. $0.01 worth) with the user ID encoded in the data field. Treat it as a signal, not the actual payment. USDC sent separately. Low-medium Fragile — two-tx flow, race conditions. Avoid.
app/api/payment/intent/route.ts The correct solution — wallet address correlation 
// STEP 1 — Before the user opens their wallet, create a server-side intent.
// Store the user's wallet address as the correlation key. No on-chain metadata needed.

export async function POST(req: Request) {
  const { userId, planId, walletAddress } = await req.json()

  // The wallet address IS the metadata — no need for on-chain encoding
  const intent = await db.paymentIntents.create({
    data: { userId, planId, walletAddress: walletAddress.toLowerCase(),
            status: 'pending', expiresAt: new Date(Date.now() + 15 * 60_000) }
  })

  return Response.json({ intentId: intent.id })
}

// STEP 2 — After on-chain verification, match payment to user via sender address
async function activateFromTransfer(transfer: TransferLog, txHash: string) {
  const senderAddress = transfer.args.from.toLowerCase()

  // Lookup the pending intent by the sender's wallet address
  const intent = await db.paymentIntents.findFirst({
    where: { walletAddress: senderAddress, status: 'pending',
             expiresAt: { gt: new Date() } }
  })
  if (!intent) throw new Error('No matching intent for sender address')

  // Activate subscription using the matched intent
  await activateSubscription(intent.userId, intent.planId, txHash)
}
⚠
Edge case — shared wallet: Two users cannot pay from the same wallet address within the same 15-minute intent window. This is an extremely rare edge case (shared wallets) but handle it by enforcing that a wallet address can only have one pending intent at a time. If a second intent is requested from the same address, invalidate the first.
IV

Shared Backend — One Verification Layer for All Three Pathways

✓
The key insight of routing everything through on-chain USDC: All three pathways end with USDC arriving at your Base wallet and a txHash that can be verified with parseEventLogs. This means verifyPayment(), the Alchemy webhook, and the idempotency logic are written once and shared. The only pathway-specific code is how the txHash reaches your backend.
lib/verifyPayment.ts Shared across all three pathways — write once 
import { createPublicClient, http, parseEventLogs, erc20Abi, parseUnits } from 'viem'
import { base } from 'viem/chains'

const client = createPublicClient({ chain: base, transport: http(process.env.ALCHEMY_BASE_RPC_URL) })

const USDC_BASE   = '0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913'
const PLAN_PRICES = { starter: parseUnits('29', 6), pro: parseUnits('79', 6) }

// Used by: P1 (/api/payment/verify), P3 (/api/webhook/nexapay), Alchemy webhook
export async function verifyUsdcPayment(hash: `0x${string}`, planId: string) {
  const receipt = await client.waitForTransactionReceipt({
    hash, timeout: 300_000, confirmations: 2  // wait for 2 blocks — reorg protection
  })
  if (receipt.status !== 'success') throw new Error('Transaction reverted')

  // ERC-20 transfers appear in logs, NOT in tx.value (which will be 0n for token transfers)
  const logs     = parseEventLogs({ abi: erc20Abi, logs: receipt.logs, eventName: 'Transfer' })
  const transfer = logs.find(l => l.address.toLowerCase() === USDC_BASE.toLowerCase()
                              && l.args.to?.toLowerCase() === process.env.BUSINESS_WALLET?.toLowerCase())
  if (!transfer) throw new Error('No USDC transfer to business wallet found in receipt')

  const expected  = PLAN_PRICES[planId]
  const actual    = transfer.args.value as bigint
  const tolerance = expected / 100n   // 1% — accounts for conversion spreads
  if (actual < expected - tolerance) throw new Error(`Underpayment: got ${actual}, expected ${expected}`)

  return { from: transfer.args.from, amount: actual }
}
